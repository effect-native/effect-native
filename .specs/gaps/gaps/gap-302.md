# Gap: Rule and Resolution Registration Mechanism

## Question

How should rules and resolutions be registered with the Engine - at Layer construction time, via a mutable registry service, or through a declarative configuration?

## Context

FR1.1.3.1 says rules are accepted "at engine initialization time" and FR1.6.1.1 says the system "shall accept rules, resolutions, and configuration at initialization."

This suggests static registration, but doesn't specify the mechanism. Options:

1. **Layer Factory Function**: `Engine.make({ rules: [...], resolutions: [...] })` returns a Layer. Rules/resolutions are baked into the layer at construction.

2. **Registry Service**: A `RuleRegistry` service with `register(rule)` methods. Engine depends on Registry. Allows dynamic registration but violates "at initialization time" if called after engine starts.

3. **Config-Driven**: Rules/resolutions defined as configuration (Effect Config or schema-validated objects), resolved at layer provision time.

4. **Effect.provide Pattern**: Rules provided as context requirements that the Engine service consumes. Each rule is its own service tag.

Key considerations:
- FR1.1.3.2 requires rejecting duplicate rule identifiers - this validation must happen somewhere
- FR1.1.3.3 requires validating rule signatures "where possible"
- NFR2.3.3 says "work with zero configuration using Engine.Default" - implies some registration happens automatically

The CrSql pattern doesn't help here since it doesn't have pluggable behaviors like rules.
