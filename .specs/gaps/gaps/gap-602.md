# Gap: Data.TaggedClass vs Schema.TaggedClass for Gap Types

## Question

Should user-defined Gap types extend `Data.TaggedClass` or `Schema.TaggedClass`? What are the implications for identity computation, serialization, and type inference?

## Context

The requirements specify gaps as `Data.TaggedClass` instances (DR4.1.1, FR1.2.1.2), but the codebase shows two patterns:

**Data.TaggedClass** (used for errors in `DebugModel.ts`):
```typescript
export class DebugStateError extends Data.TaggedError("DebugStateError")<{ ... }> {}
```

**Schema.TaggedClass** (used in `openrouter/src/index.ts`):
```typescript
export class OpenRouterError extends Schema.TaggedError<OpenRouterError>()("OpenRouterError", { ... }) {}
```

Key differences:
- `Schema.TaggedClass` provides built-in JSON serialization (needed for FR1.2.2.5, DR4.2.2)
- `Schema.TaggedClass` allows field annotations for validation
- `Data.TaggedClass` is simpler, implements Equal/Hash by default
- Schema classes can declare identity-contributing fields via annotations (gap-101)

The choice affects:
- How gap identity is computed (FR1.2.2)
- Work order serialization (DR4.2.2)
- User ergonomics when defining custom gap types (SC7.2.1)
- Whether we need a separate serialization layer

Related requirements: FR1.2.1, FR1.2.2.5, DR4.1.1, DR4.2.2
