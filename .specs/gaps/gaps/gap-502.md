# Gap: Gap Status Lifecycle and Transition Rules

## Question

What are the valid state transitions for gap status, and how are transitions enforced and tracked?

## Context

FR1.2.4.1-4 define three gap statuses (`unresolved`, `escalated`, `resolved`) but don't fully specify the transition model:

**Ambiguous transitions:**
- Can a gap transition from `escalated` back to `unresolved` if re-detected in a later iteration?
- Can a `resolved` gap become `unresolved` again if re-detected (indicating resolution didn't actually work)?
- Is `resolved` a terminal state within a single run, or can it be revisited?

**Tracking concerns:**
- Should we track transition history (for debugging/auditability)?
- Does gap identity persist across status changes, or does status change create a "new" gap instance?
- How do we handle the "same" gap detected in iteration N, resolved, then re-detected in iteration N+1?

**Semantic impact:**
- FR1.5.2.1 says loop terminates when "no new actionable gaps" detected—does a re-detected previously-resolved gap count as "new"?
- This directly affects loop termination semantics

This is critical for correctness: incorrect transition handling could cause infinite loops or premature termination.

Related requirements: FR1.2.4.1-4, FR1.5.1.4, FR1.5.2.1-2
