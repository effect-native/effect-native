# Gap: Loop State Structure and Ownership

## Question

What is the concrete structure of the reconciliation loop state, and where does it live during execution?

## Context

FR1.5.1.1-4 describe an iterative loop that must track:
- Current iteration count
- All gaps detected across iterations
- Gap status transitions (unresolved → resolved/escalated)
- Which gaps are "new" vs "carried over" (FR1.5.1.4)
- Deduplication state

The requirements specify "in-memory store for gap tracking" (FR2.2.3) but don't clarify:

1. **State ownership**: Is loop state held in a `Ref`, passed as accumulator through recursion, or managed by a service?
2. **State structure**: Is this a single monolithic state object or distributed across multiple refs/services?
3. **State isolation**: How do we ensure loop state doesn't leak between reconciliation runs?
4. **State access**: Who can read/write loop state—only the engine, or can rules/resolutions inspect it?

This is blocking because Effect-TS offers multiple patterns (Ref, FiberRef, Service state, functional accumulator) with different tradeoffs for testability, composability, and safety.

Related requirements: FR1.5.1.1-4, FR2.2.3, NFR2.4.4, TC3.1.5
