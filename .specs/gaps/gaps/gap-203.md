# Gap: Engine Configuration and Builder API

## Question

What is the API for constructing and configuring an Engine instance?

FR1.6.1.1 states the system accepts rules, resolutions, and configuration at initialization. But the shape of this API is undefined:

1. **Builder pattern**: `Engine.builder().withRule(r1).withResolution(h1).withConfig({maxIterations: 5}).build()`
2. **Config object**: `Engine.make({ rules: [r1], resolutions: [h1], config: {...} })`
3. **Layer composition**: `Engine.layer.pipe(Layer.provide(Rule.layer(r1)), Layer.provide(Resolution.layer(h1)))`
4. **Fluent registration**: `Engine.make().register(Rule.define(...)).register(Resolution.define(...))`

Each has different implications for:
- Type safety (can the compiler ensure all gap types have resolutions?)
- Ergonomics (how many lines of code to set up?)
- Flexibility (can configuration change after creation?)

## Context

FR2.2.1 requires an `Engine.Default` layer with sensible defaults, and FR2.3.1 allows individual services to be overridden via Layer composition. This suggests Layer composition is part of the API.

But rules and resolutions aren't "services" in the Effect sense - they're domain objects. How do they get registered?

SC7.2.1 requires "Tom shall be able to write and register a new rule in under 5 minutes" - the registration ergonomics directly affect this.
