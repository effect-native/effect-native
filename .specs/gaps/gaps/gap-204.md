# Gap: Rule Context Type and Service Dependencies

## Question

What is the type of the `context` parameter passed to rules, and how do rules declare their service dependencies?

FR1.1.1.1 specifies rules have signature `(context: Context) => Effect<Gap[], RuleError, R>`. But:

1. What is `Context`? Is it a user-defined type, a generic parameter, or a fixed structure?
2. The `R` type parameter suggests rules can require Effect services (like `FileSystem`). How is this expressed in the rule definition?
3. If different rules need different services, how does the Engine know what to provide?

Options:
- Fixed `Context` with common fields (clock, logger) + generic `R` for additional services
- Generic `Context<R>` where rules fully define their needs
- No `Context` at all - rules use `Effect.serviceWith` internally

## Context

FR1.1.1.3 says "The system shall pass a single context object containing all dependencies to each rule" - implying a structured context rather than pure Effect service injection.

But Effect-TS convention is to use services + Layers for dependency injection. Having a separate "context" object creates two DI systems.

This affects:
- Rule portability (can rules be tested without the Engine?)
- Service discovery (what services are available?)
- Type inference complexity

NFR2.4.1 requires "rules to be tested in isolation without the engine" - the context design directly affects testability.
