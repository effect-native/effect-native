# Gap: Timeout Implementation Pattern

## Question

How should timeouts be implemented for rules and resolutions, and what error type do they produce?

The requirements mention timeouts:
- FR1.1.2.4: Rule timeout terminates rule execution and logs timeout error

But the design must specify:
1. Is timeout a configuration per-rule, global, or both (with per-rule override)?
2. What Effect pattern implements timeout? `Effect.timeout`? `Effect.timeoutFail`? `Effect.interruptible`?
3. Does timeout produce a specific error type (`RuleTimeoutError`) or use Effect's built-in timeout handling?
4. Can a rule "catch" its own timeout and do cleanup? Or is it hard-interrupted?
5. Should resolutions also have timeouts? (Not mentioned in requirements)

Effect's timeout returns `Option.none` on timeout, while `timeoutFail` returns a custom error. The choice affects error handling downstream.

## Context

Timeouts interact with several requirements:
- FR1.1.2.3: Rule errors should be logged with rule identifier
- NFR2.2.4: Isolate rule failures so one failing rule doesn't affect others
- NFR2.1.1: Single rule evaluation under 100ms (implies timeout should be configurable above this)

The failure lens suggests timeout should be "LOUD" - operators need to know something hung.

Key design decision: Should timeout be modeled as:
A. A variant of `RuleError` (e.g., `RuleError.Timeout`)
B. A separate error type (`TimeoutError`) that wraps the timed-out operation
C. Effect's built-in `TimeoutException` (not a tagged error)

Option A is most consistent with the typed error philosophy. Option C is easiest but loses type information.

Also: What's a sensible default timeout? 5 seconds? 30 seconds? No default (require explicit config)?
