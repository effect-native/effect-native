# Gap: Layer Composition Strategy for Engine.Default and Engine.Test

## Question

How should Engine.Default and Engine.Test layers be structured to allow selective service replacement while maintaining type safety?

## Context

FR2.2.1 requires `Engine.Default` with sensible defaults, FR2.3.3 requires `Engine.Test` for testing, and FR2.3.1-2 require individual services to be overridable.

Effect offers several Layer composition patterns:

**Pattern A: Merged defaults with provideMerge**
```typescript
Engine.Default.pipe(Layer.provideMerge(CustomWorkOrderSink.layer))
```

**Pattern B: Layer.provide with explicit dependencies**
```typescript
Layer.provide(Engine.layer, Layer.merge(WorkOrderSink.Default, GapStore.Default))
```

**Pattern C: Effect.Service with dependencies array**
```typescript
class Engine extends Effect.Service<Engine>()("...", {
  dependencies: [WorkOrderSink.Default, GapStore.Default],
  effect: Effect.gen(function*() { ... })
}) {}
```

Key questions:
1. Should `Engine.Default` be a single opaque layer, or a composition of sub-service layers?
2. How do we type `Engine.Test` to expose inspection methods (NFR2.4.4)?
3. Should custom services merge with or replace defaults (FR2.3.2 says "prefer custom")?
4. How do we handle the `R` type parameter when rules require additional services?

The choice affects whether users can easily swap WorkOrderSink without understanding internal structure.

Related requirements: FR2.2, FR2.3, NFR2.4.4
