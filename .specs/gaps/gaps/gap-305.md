# Gap: WorkOrderSink Service Contract and Error Handling

## Question

What is the complete service contract for WorkOrderSink, including retry behavior ownership and error type propagation?

## Context

FR2.1.4 defines `write(workOrder: WorkOrder) => Effect<void, WorkOrderError>`. FR1.4.3.3 says "retry up to 3 times with exponential backoff" on failure, and FR1.4.3.4 says after retries fail, "log critical error and mark gap as unresolved."

Unclear ownership:
1. **Retry Logic Location**: Does the WorkOrderSink implementation handle retries internally, or does the Engine/Escalator wrap calls with retry logic?
   - If internal: Custom implementations must implement retry. Harder for implementers.
   - If external: Engine handles retry. WorkOrderSink is simple. But custom impls can't customize retry behavior.

2. **Error Type**: Is `WorkOrderError` a specific tagged error class, or a union of possible errors? Should it include retry-exhaustion as a distinct case?

3. **Idempotency**: If retries happen, must WorkOrderSink.write be idempotent? FR2.1.4 doesn't specify. The filesystem default (FR2.1.2) could use work order ID as filename for natural idempotency.

4. **Async Acknowledgment**: Does `write` completing mean the work order is durably persisted, or just accepted? This affects the retry/failure semantics.

This is the primary extension point (FR2.1.3 allows custom implementations), so the contract must be precise.
