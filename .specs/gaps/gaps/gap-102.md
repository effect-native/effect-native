# Gap: Type Parameter Threading Through Rule/Resolution Signatures

## Question

How should the generic type parameters for context dependencies (R), errors (E), and gap types flow through the Rule, Resolution, and Engine type signatures?

## Context

FR1.1.1.1 specifies rules as `(context: Context) => Effect<Gap[], RuleError, R>` and FR1.3.1.1 specifies resolutions as `(gap: Gap) => Effect<void, ResolutionError, R>`.

Key design decisions:
1. Should `R` (requirements) be inferred per-rule or declared engine-wide?
2. How does the engine aggregate multiple rules with different `R` requirements?
3. Should gap types be parameterized on the rule that produces them?
4. How do we ensure type-safe matching between gaps and their resolutions?

Effect-TS patterns to consider:
- Union of requirements via `Effect.all` inference
- Explicit service tags vs. inferred dependencies
- Branded types for gap-resolution pairing

This directly impacts API ergonomics and type safety guarantees.

Related requirements: FR1.1.1.1, FR1.3.1.1, TC3.1.4, NFR2.3.4
