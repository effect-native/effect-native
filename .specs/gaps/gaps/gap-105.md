# Gap: Error Type Hierarchy and Tagged Error Design

## Question

What is the complete error type hierarchy, and how should errors relate to their originating contexts (rules, resolutions, escalation)?

## Context

Requirements reference multiple error types without defining their relationship:
- `RuleError` (FR1.1.1.1)
- `ResolutionError` (FR1.3.1.1)  
- `WorkOrderError` (FR2.1.4)
- Timeout errors (FR1.1.2.4)
- Configuration/validation errors (FR1.6.1.3)

Design decisions needed:
1. Should there be a base `GapsError` type that all errors extend?
2. How to associate errors with their source (rule id, gap identity, etc.)?
3. Which errors are "expected" (part of E channel) vs. "defects" (should never happen)?
4. Should errors include structured context or just messages?
5. How to type error recovery paths (e.g., rule failure continues, resolution failure escalates)?

Effect-TS patterns: `Data.TaggedError`, cause chains, `Cause.fail` vs `Cause.die`

This affects all error handling paths and the API contract for custom implementations.

Related requirements: FR1.1.1.1, FR1.3.1.1, FR2.1.4, FR1.1.2.3-4, IR5.1.2
