# Gap: Error Channel vs Defect Boundary

## Question

Where is the boundary between recoverable errors (Error channel) and unrecoverable defects (Cause.Die)?

In Effect-TS:
- **Error channel (`E`)**: Expected failures that callers must handle
- **Defects (`Cause.Die`)**: Unexpected failures (bugs, invariant violations) that should crash

The requirements imply certain failures are "expected":
- No resolution matches a gap (FR1.3.4.1) - escalate, don't crash
- Rule times out (FR1.1.2.4) - log and continue
- Work order write fails (FR1.4.3.3) - retry, then mark unresolved

But others seem like defects:
- Gap identity computation fails on valid gap data
- Resolution registered for non-existent gap type
- Internal state corruption

The question: Should rule/resolution authors be able to return `Effect.die()` to signal "this is a bug, not a gap"? Or should all rule/resolution failures flow through the Error channel?

## Context

This matters because:
1. Defects propagate differently - they bypass error handlers
2. Defects are harder to test
3. The "never silently lose a gap" requirement (NFR2.2.1) implies even defects must be captured

Effect best practice: Use defects sparingly, only for true invariant violations. But the reconciliation loop needs to be resilient - a rule author's bug shouldn't crash the entire loop.

Options:
A. All rule/resolution errors go through Error channel; defects only for engine internals
B. Allow rules/resolutions to signal defects; engine catches and converts to errors
C. Strict separation: rules/resolutions can only return errors; defects are always engine bugs

Requirement FR1.1.2.3 says "IF a rule throws an error THEN the system shall log the error... and continue" - but "throws" is ambiguous. Does this include `Effect.die()`?
