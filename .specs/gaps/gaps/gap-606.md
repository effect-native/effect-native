# Gap: Module Organization Following effect-native Conventions

## Question

How should the gaps package organize its modules to follow effect-native conventions (public API re-exports, internal implementations, model types)?

## Context

The codebase shows a clear organizational pattern:

**Public module** (`Thing.ts`):
```typescript
export const TypeId: typeof internal.TypeId = internal.TypeId
export interface Thing<A> extends internal.Thing<A> {}
export const make: typeof internal.make = internal.make
```

**Internal module** (`internal/thing.ts`):
```typescript
/** @internal */
export const TypeId: unique symbol = Symbol.for("@effect-native/patterns/Thing")
/** @internal */
export const make = <A>(input: ThingInput<A>): Thing<A> => { ... }
```

**Model module** (`DebugModel.ts`):
- Defines interfaces, types, errors, and Context.Tags
- No implementation logic, just shapes

For gaps, we need to organize:
- `Gap` type and related types (GapStatus, identity types)
- `Rule`, `Resolution` types
- `Engine` service and sub-services
- `WorkOrder`, `ReconciliationResult` types
- Error types
- Internal implementations (loop logic, deduplication, identity computation)

Questions:
1. Should we have `GapModel.ts` + `Gap.ts` + `internal/gap.ts` pattern?
2. Where do rule/resolution registration types live?
3. How does the index.ts barrel export organize these?
4. Should we use TypeId symbols for Gap types (for runtime type checking)?

Related requirements: TC3.1.4 (strict TypeScript), NFR2.3.4 (types for public APIs)
