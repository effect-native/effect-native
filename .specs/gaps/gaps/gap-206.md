# Gap: WorkOrderSink Service Interface Design

## Question

What is the complete interface for the `WorkOrderSink` service, and should there be additional sink types?

FR2.1.4 defines only `write(workOrder: WorkOrder) => Effect<void, WorkOrderError>`. But:

1. Should there be batch operations: `writeMany(workOrders: WorkOrder[]) => Effect<void, WorkOrderError>`?
2. Should there be query operations: `read(id: string) => Effect<WorkOrder | null, WorkOrderError>`?
3. Should the filesystem sink have configuration (path, format)?
4. What fields should `WorkOrderError` have?

Additionally, should there be other sink types beyond filesystem?
- In-memory sink for testing (FR2.3.3 mentions `Engine.Test` layer)
- HTTP sink for remote escalation
- Composite sink (write to multiple destinations)

## Context

FR2.1.2 specifies a default filesystem implementation, and FR2.1.3 allows custom implementations via Layer composition.

The interface design affects:
- How easily users can implement custom sinks
- Whether the default sink is sufficient for production use
- Integration with external systems (ticketing, alerting)

SC7.2.3 requires "Bramwell shall be able to view escalated work orders in the filesystem" - the filesystem sink must produce human-readable output, which implies format decisions (JSON, Markdown, YAML?).
