# Gap: Immutability Guarantees for Gap Data

## Question

Should gap instances be immutable after creation, and how do we enforce this while supporting status transitions?

## Context

TC3.1.5 mandates "no global mutable state" and TC3.1.6 requires "no side effects outside Effect's control." However, gaps need status tracking (FR1.2.4) which implies state changes.

**Design tension:**
- `Data.TaggedClass` from Effect-TS produces structurally immutable objects
- But we need to track status per gap (`unresolved` → `resolved` → etc.)
- We need to track metadata (which iteration detected it, resolution attempts, etc.)

**Options:**
1. **Immutable gaps, mutable registry**: Gaps are immutable; status is tracked in a separate Map<GapIdentity, GapStatus>
2. **Immutable with copy-on-write**: Each status change creates a new gap instance with updated status
3. **Wrapper pattern**: Immutable gap data wrapped in a mutable TrackedGap container
4. **Full immutability**: All state changes produce new state snapshots (functional approach)

**Implications:**
- Affects memory usage (copy-on-write with 1000 gaps)
- Affects identity semantics (is gap-with-status-A === gap-with-status-B?)
- Affects how rules receive gap data (do they see status?)
- Affects test assertions (structural equality vs identity)

Related requirements: FR1.2.4.1-4, TC3.1.5, TC3.1.6, DR4.1.1-3, NFR2.1.3
