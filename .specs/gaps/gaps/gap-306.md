# Gap: Service Lifecycle and Scoping

## Question

What is the expected lifecycle/scope of Engine and its dependent services - per-reconciliation-run, long-lived singleton, or caller-controlled?

## Context

The requirements describe both:
- Long-lived concepts: "reconciliation loop" (FR1.5), "iteration count" (FR1.5.2.3)
- Per-run concepts: "within a reconciliation run" (FR2.2.3), "unique within a reconciliation run" (DR4.2.3)

Scope implications:

1. **Engine Instance Scope**: Is the Engine a singleton service that processes multiple `reconcile()` calls over its lifetime, or is a new Engine instance created per reconciliation run?

2. **GapStore Scope**: FR2.2.3 says "in-memory store for gap tracking within a reconciliation run." Does the store reset between runs? If Engine is long-lived, how is per-run state isolated?

3. **Service Acquisition**: When a rule runs, are services acquired fresh each time or shared across the rule execution? This affects resource management for rules that use external services.

4. **Finalizers**: The CrSql pattern uses `Effect.addFinalizer` for cleanup. Should Engine services have finalizers? What cleanup is needed?

Effect patterns:
- `Layer.scoped` for per-request resources
- `Effect.acquireRelease` for managed lifecycles
- `Ref` for mutable state within a scope

The design must clarify whether state is scoped to Engine lifetime or reconciliation run lifetime.
