# Gap: Rule and Resolution Function Signatures with Effect.gen vs Pipe

## Question

Should rule and resolution function signatures use `Effect.gen` style internally, and how should the type parameter `R` (dependencies) be exposed to users?

## Context

FR1.1.1.1 specifies rules as `(context: Context) => Effect<Gap[], RuleError, R>` and FR1.3.1.1 specifies resolutions as `(gap: Gap) => Effect<void, ResolutionError, R>`.

The codebase predominantly uses `Effect.gen` (see `openrouter/src/index.ts:36`):
```typescript
const callModel = Effect.fn(function*<const TTools extends ...>(request: ...) {
  const result = yield* Effect.try({ ... })
  ...
})
```

Design questions:
1. Should we require users to write rules/resolutions using `Effect.gen`, or support both styles?
2. How do we type the `R` parameter to allow rules to access services (e.g., filesystem, HTTP)?
3. Should we provide a `Rule.make` / `Resolution.make` helper that wraps user functions?
4. How do we ensure proper type inference when rules require different services?

Example concern - if Rule A needs `FileSystem` and Rule B needs `HttpClient`, the Engine must provide both via Layers. How is this typed?

Related requirements: FR1.1.1.1, FR1.1.1.3, FR1.3.1.1, TC3.1.4
