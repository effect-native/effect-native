# Gap: Work Order Schema and Extensibility

## Question

What is the complete schema for work orders, and how should it support extension for different escalation targets beyond filesystem?

## Context

DR4.2.1 specifies work orders contain: id, gap type, gap data, rule id, timestamp, and optional resolution failure. DR4.2.2 requires JSON serializability.

Open questions:
1. How is "gap data" serialized when gaps may contain non-serializable fields (functions, Effects, circular refs)?
2. Should work orders include the full gap or just identity + serializable subset?
3. What format for "resolution failure" - just message, or structured error with stack?
4. Should work orders be versioned for future schema evolution?
5. How to represent optional fields (resolution failure) in TypeScript - `| undefined` vs discriminated union?

Additionally, FR2.1.3 allows custom WorkOrderSink implementations - the schema must be defined such that custom sinks (e.g., HTTP API, queue) receive consistent data.

Related requirements: DR4.2.1, DR4.2.2, DR4.2.3, FR2.1.4, FR1.4.2.1-4
