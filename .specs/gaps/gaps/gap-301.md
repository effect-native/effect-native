# Gap: Engine Service Interface Granularity

## Question

Should the Engine be a single monolithic service with all operations, or should it be decomposed into smaller focused services (RuleExecutor, GapStore, ResolutionExecutor, EscalationService)?

## Context

FR2.3.1 states "The system shall allow individual services to be overridden via Effect Layer composition." This implies services should be swappable. However, the requirements don't clarify the service boundary granularity.

Options:
1. **Monolithic Engine**: Single `GapsEngine` service with all methods. Simple DI but limited override granularity.
2. **Decomposed Services**: Separate `RuleExecutor`, `GapStore`, `ResolutionMatcher`, `Escalator` services. The Engine composes these. More flexible but more complex layer wiring.
3. **Hybrid**: Core Engine service with pluggable strategy services (WorkOrderSink is already called out explicitly in FR2.1).

The choice affects:
- Testability (FR2.4 requires isolated testing)
- Custom implementation effort (SC7.2.1 - write rule in under 5 mins)
- Layer composition complexity for callers

The existing `CrSql` pattern uses a single service with many methods, but it's purely data-oriented. The Engine has distinct behavioral phases (detect, resolve, escalate) that might benefit from separation.
