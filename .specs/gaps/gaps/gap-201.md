# Gap: Module Organization and Entry Point Structure

## Question

Should the package expose a single namespace entry (`import { Gaps } from "@effect-native/gaps"`) or multiple named exports (`import { Engine, Rule, Gap, Resolution } from "@effect-native/gaps"`)?

The monorepo shows two patterns:
1. **Namespace pattern** (`@effect-native/patterns`): `export * as Thing from "./Thing.js"`
2. **Direct exports pattern** (`@effect-native/fetch-hooks`): Individual function/type exports

Which pattern should `@effect-native/gaps` use, and what modules should exist?

## Context

FR1.1-FR1.6 define several core concepts: Rules, Gaps, Resolutions, Escalation, Reconciliation Loop, and Engine Lifecycle. Each could be its own module or they could be combined.

NFR2.3.4 requires TypeScript types for all public APIs, and SC7.2.1 requires that "Tom shall be able to write and register a new rule in under 5 minutes" - suggesting the import ergonomics matter significantly.

The Effect ecosystem convention is typically namespace exports (e.g., `Effect.gen`, `Layer.provide`), but this package is smaller and more focused than core Effect.

Options:
- `import { Engine, Rule, Gap } from "@effect-native/gaps"` (flat)
- `import { Gaps } from "@effect-native/gaps"; Gaps.Engine.reconcile(...)` (namespace)
- `import * as Engine from "@effect-native/gaps/Engine"` (per-module imports)
