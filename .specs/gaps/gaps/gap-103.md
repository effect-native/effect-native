# Gap: Internal Gap State Representation vs. User-Facing Gap Type

## Question

Should internal gap tracking state (status, iteration detected, resolution attempts) be stored separately from the user-defined gap data, or wrapped in an envelope type?

## Context

Requirements specify:
- Gaps are `Data.TaggedClass` instances (DR4.1.1)
- Gaps have status: unresolved/escalated/resolved (FR1.2.4.1)
- System tracks which gaps are new vs carried over (FR1.5.1.4)
- Results include categorized gaps by status (FR1.5.3.1-4)

Design tension:
- User-defined gaps should be simple domain objects
- Engine needs metadata (status, first-seen iteration, resolution attempts, etc.)
- Identity computation should not include metadata
- Serialization concerns differ between domain data and tracking state

Options:
1. Envelope type: `TrackedGap<G extends Gap> = { gap: G, status: Status, ... }`
2. Mixin: `Gap & { _status: Status, _iteration: number, ... }`
3. Parallel data structure: `Map<GapIdentity, GapMetadata>`

This affects the entire internal API surface and result types.

Related requirements: DR4.1.1, FR1.2.4.1-4, FR1.5.1.4, FR1.5.3.1-4
