# Gap: Default vs Test Layer Composition Strategy

## Question

What is the precise Layer composition strategy for Engine.Default vs Engine.Test, and how should custom service overrides merge with these base layers?

## Context

FR2.2.1 requires `Engine.Default` with "sensible defaults" and FR2.3.3 requires `Engine.Test` with "in-memory implementations." FR2.3.2 says custom layers should "merge with defaults, preferring custom implementations."

Unclear aspects:

1. **Layer Dependency Graph**: Does Engine.Default depend on platform services (FileSystem from @effect/platform)? If so, does Engine.Test mock these, or does it provide a completely separate implementation tree?

2. **Override Granularity**: Can users override just WorkOrderSink while keeping default Logger, Clock, and GapStore? What's the composition API?

3. **Test Layer Inspection**: NFR2.4.4 says "Engine.Test shall allow inspection of internal state after reconciliation." This implies the test layer exposes additional methods not in the production interface. How is this modeled - separate service interface, or augmented implementation?

Options for override composition:
- `Layer.merge(Engine.Default, customWorkOrderSink)` - requires service tags to be well-defined
- `Engine.make({ workOrderSink: customImpl })` - factory function with optional overrides
- `Engine.Default.pipe(Layer.provide(customWorkOrderSink))` - Effect's standard layer provision

The CrSql codebase uses `Layer.effect` and `Layer.succeed` patterns but doesn't demonstrate complex override composition.
