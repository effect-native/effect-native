# Gap: Rule Context Service Dependencies

## Question

How should rules access services they depend on - via a generic Context bag, through explicit service requirements in their Effect type, or through a provided context parameter?

## Context

FR1.1.1.1 specifies rules have signature `(context: Context) => Effect<Gap[], RuleError, R>`. The `R` type parameter suggests rules can have arbitrary service requirements. FR1.1.1.3 says "pass a single context object containing all dependencies."

These requirements appear contradictory:
- The `R` type implies rules declare their dependencies in the Effect type and get them via `yield* Service`
- The "context object" language implies passing dependencies as a function argument

Options:

1. **Context Parameter Only**: `(ctx: { fileSystem: FileSystem, config: Config }) => Effect<Gap[], RuleError, never>`. Rules receive dependencies as data, no service requirements in R.

2. **Effect Services Only**: `() => Effect<Gap[], RuleError, FileSystem | Config>`. Rules use standard Effect DI. The "context" in requirements is just the Effect context.

3. **Hybrid - Context Data + Optional Services**: `(ctx: RuleContext) => Effect<Gap[], RuleError, R>`. RuleContext contains common data (current iteration, timestamps). Rules can also require services in R for advanced use cases.

This affects:
- Rule authoring ergonomics (SC7.2.1 - 5 minute rule creation)
- Layer composition (how to satisfy rule requirements)
- Testability (NFR2.4.1 - test rules in isolation)
