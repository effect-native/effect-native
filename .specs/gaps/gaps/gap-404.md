# Gap: Recovery Strategy for Escalation Failures

## Question

What happens when the escalation mechanism itself fails, and how do we prevent gaps from being silently lost?

The requirements specify (FR1.4.3.3-4):
- Retry work order creation up to 3 times with exponential backoff
- After retries exhausted, log critical error and mark gap as unresolved

But this raises further questions:
1. What state is a gap in after escalation failure? Still `unresolved`? A new `escalation_failed` status?
2. Does the gap remain in the loop for the next iteration, or is it "parked"?
3. If the WorkOrderSink is persistently unavailable, do we keep retrying every iteration forever?
4. Should there be a circuit breaker pattern for the escalation path?
5. What's the escalation path for escalation failures? (Meta-escalation to logs? Stdout? File dump?)

The failure lens says "Escalation must be infallible from the user's perspective." But the implementation must handle the reality that the sink can fail.

## Context

This is the "fallback's fallback" problem. The architecture assumes Bramwell (via WorkOrderSink) is always available. But:
- Filesystem could be full
- Network sink could be unreachable  
- Custom sink implementation could be buggy

The principle "Silent failures are catastrophic" means we need an answer for: "What if we literally cannot tell anyone about this gap?"

Options:
A. Log to stderr as last resort (always available)
B. Accumulate failed escalations in memory, dump on shutdown
C. Fail the entire loop with `reason: "error"` if escalation is impossible
D. Park gaps in a local file as "dead letter queue"

Requirement NFR2.2.1 ("never silently lose a detected gap") is the key constraint.
