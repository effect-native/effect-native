# Gap: Error Context and Traceability

## Question

What contextual information should errors carry, and how do we maintain traceability through the reconciliation loop?

The requirements demand rich error context:
- FR1.1.2.3: Log error with rule identifier
- FR1.4.2.2: Work order includes gap type, gap data, originating rule identifier, timestamp
- FR1.4.2.3: Include resolution failure reason in work order
- NFR2.3.1: Clear error messages identifying the source of failures

This implies errors need to carry:
- Origin (which rule/resolution produced this error)
- Timing (when did it occur, during which iteration)
- Subject (which gap was being processed)
- Causation (what upstream error caused this, if any)

Questions:
1. Should errors embed context directly, or should we use Effect's Span/annotation system?
2. How do we correlate errors across the loop (e.g., "rule X failed at iteration 3, then again at iteration 5")?
3. Should there be a correlation ID per reconciliation run for log aggregation?
4. How much context is "enough" vs "too verbose" for work orders?

## Context

Effect provides several mechanisms:
- `Effect.annotateLogs` for adding context to log spans
- `Effect.withSpan` for tracing
- Custom error types with data fields
- `Cause` which preserves the full error tree

The tension is between:
- **Rich context**: Great for debugging, but adds complexity and payload size
- **Minimal context**: Simpler, but "clear error messages" requirement is hard to meet

The failure lens emphasizes that errors should be "actionable." Bramwell receiving a work order with just "ResolutionError" is useless. Bramwell receiving "ResolutionError: RestartService failed for gap ServiceDown(service='api-gateway') because SSH connection refused to host 10.0.0.5" is actionable.

Pattern question: Do we use Effect's `Cause.annotated` or define our own context-carrying error types?
