# Gap: Error Propagation in the Reconciliation Loop

## Question

How do errors propagate through the reconciliation loop, and at what granularity do we catch/handle them?

The loop has nested structure:
```
Loop iteration
  -> Rule evaluation (N rules)
    -> Gap collection
  -> Resolution execution (M gaps)
    -> Escalation (on failure)
```

Questions:
1. If one rule fails, do we abort gap collection or continue with remaining rules?
2. If one resolution fails, do we immediately escalate or batch escalations?
3. If escalation fails (work order write), do we retry immediately or at end of iteration?
4. What errors cause the entire loop to terminate with `reason: "error"`?

The requirements give partial answers:
- FR1.1.2.3: Continue executing remaining rules after rule failure
- FR1.3.3.3: Don't retry resolution automatically
- FR1.4.3.3: Retry escalation up to 3 times

But the orchestration pattern isn't specified. Do we use:
- `Effect.forEach` with error accumulation?
- `Effect.all` with `{ mode: "either" }`?
- Manual try/catch with result aggregation?

## Context

This is critical for the "loud failures are recoverable" principle. The loop must:
1. Never silently drop gaps (NFR2.2.1)
2. Isolate rule failures (NFR2.2.4)
3. Report accurate termination reason (NFR2.2.2)

Effect provides tools like `Effect.catchAll`, `Effect.catchTag`, `Effect.either`, and `Effect.partition`. The design must specify which pattern to use where.

Edge case: What if ALL rules fail? Is that `reason: "error"` or `reason: "stable"` (no gaps detected because we couldn't run rules)?
