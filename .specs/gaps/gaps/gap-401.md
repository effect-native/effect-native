# Gap: Error Type Taxonomy

## Question

What is the complete taxonomy of tagged error types, and how should they be organized in the type hierarchy?

The requirements mention several error categories implicitly:
- `RuleError` (FR1.1.1.1)
- `ResolutionError` (FR1.3.1.1)
- `WorkOrderError` (FR2.1.4)

But the design needs to specify:
1. Are these separate union types or a single `GapsError` union?
2. What specific variants exist within each category (e.g., `RuleTimeoutError`, `RuleExecutionError`, `RuleDependencyError`)?
3. Should errors carry structured context (rule ID, gap identity, timestamps) or just messages?
4. How do we distinguish "expected" errors (no resolution found) from "unexpected" errors (rule threw exception)?

## Context

Effect-TS strongly encourages modeling errors as tagged discriminated unions in the error channel. This provides:
- Exhaustive pattern matching at call sites
- Type-safe error narrowing
- Clear documentation of failure modes

The requirements state "Silent failures are catastrophic. Loud failures are recoverable." This implies errors need to carry enough context to be actionable. But over-engineering the error hierarchy adds complexity.

Key requirements impacted:
- FR1.1.2.3: Rule errors must be logged with rule identifier
- FR1.3.1.4: Resolution failures must be logged and trigger fallback
- FR1.4.2.3: Resolution failure reason must be included in work orders
- NFR2.3.1: Clear error messages identifying the source of failures

The failure lens analysis identifies 8 failure categories. Not all need distinct types, but the mapping must be explicit.
