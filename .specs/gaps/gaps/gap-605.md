# Gap: Tagged Error Hierarchy for RuleError, ResolutionError, and WorkOrderError

## Question

How should we structure the error type hierarchy to distinguish rule failures, resolution failures, escalation failures, and timeout errors while maintaining Effect-TS error handling idioms?

## Context

The requirements mention several error scenarios:
- Rule throws error (FR1.1.2.3)
- Rule timeout (FR1.1.2.4)
- Resolution fails (FR1.3.1.4, FR1.3.3.4)
- Work order creation fails (FR1.4.3.3-4)
- Fatal errors causing loop termination (FR1.5.2.4)

Effect-TS pattern from `DebugModel.ts` defines a union of tagged errors:
```typescript
export type DebugError = DebugStateError | DebugTransportError | DebugCommandError | ...
```

Design questions:
1. Should we define a single `GapsError` union, or separate error channels?
2. How do rule-specific errors (user code) compose with engine errors (our code)?
3. Should timeout errors be a subtype or a wrapper with the underlying error?
4. How do we ensure FR1.1.2.3 (log and continue) vs FR1.5.2.4 (terminate) distinction?
5. Should errors carry structured context (rule ID, gap ID) or just messages?

The error structure affects:
- Whether callers can pattern-match on specific failure types
- How errors are logged (NFR2.3.1)
- Whether errors are recoverable or terminal

Related requirements: FR1.1.2.3-4, FR1.3.1.4, FR1.4.3.4, FR1.5.2.4, NFR2.3.1
