# Gap: Reconciliation Result Type and Access Patterns

## Question

What is the exact shape of the `ReconciliationResult` type, and what accessors/utilities should be provided?

FR1.5.3.1-4 and DR4.3.1-3 specify the result contains: termination reason, iteration count, resolved gaps, escalated gaps, and unresolved gaps. But the API for working with this result is undefined:

1. Should gaps be grouped by status in separate arrays, or a single array with status field?
2. Should there be helper methods like `result.hasUnresolvedGaps()`, `result.getGapsByType("MissingFile")`?
3. Should the result include timing information (duration per iteration, per rule)?
4. How should the error case (DR4.3.3) be represented - union type, separate error field, or Effect failure?

## Context

SC7.2.4 requires "Agents shall be able to programmatically invoke reconciliation and inspect results" - the result type is the primary interface for this.

The result needs to support:
- Quick "did it stabilize?" checks
- Detailed inspection of what happened
- Serialization for logging/storage
- Type-safe access to gap data

NFR2.3.1 requires "clear error messages identifying the source of failures" - error reporting is part of the result API.
