# Gap: Service Definition Pattern for Engine and Sub-services

## Question

Should we use `Effect.Service` class pattern (as in OpenRouter) or `Context.GenericTag` + separate interface pattern (as in Debug) for defining the Engine, WorkOrderSink, and other services?

## Context

The codebase shows two distinct patterns for Effect services:

**Pattern A: Effect.Service class** (`openrouter/src/index.ts`):
```typescript
export class OpenRouter extends Effect.Service<OpenRouter>()("@effect-native/openrouter/OpenRouter", {
  dependencies: [...],
  effect: Effect.gen(function*() { ... })
}) { ... }
```

**Pattern B: Context.GenericTag + interface** (`debug/src/DebugModel.ts`):
```typescript
export interface Service { ... }
export const Debug = Context.GenericTag<Service>("@effect-native/debug/Debug")
```

For gaps, we need:
- `Engine` service (FR1.6, FR2.2)
- `WorkOrderSink` service (FR2.1)
- Potentially `GapStore` for tracking gap state (FR2.2.3)

The choice affects:
- How `Engine.Default` and `Engine.Test` layers are composed (FR2.2.1, FR2.3.3)
- Type inference for service dependencies in rules and resolutions
- Whether we can use `Effect.Service.Default` auto-generated layers
- Testing ergonomics (NFR2.4)

Related requirements: FR2.1, FR2.2, FR2.3, NFR2.4
