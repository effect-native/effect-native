# Gap: Gap Type Definition Mechanism

## Question

How should users define custom Gap types that work with the type system for resolution matching?

FR1.2.1.2 requires gaps as `Data.TaggedClass` instances, and FR1.3.2.1 requires resolution matching by exact `_tag` match. But the API for defining these is unclear:

1. Should there be a helper function like `Gap.define("MissingFile", { path: Schema.String })`?
2. Should users extend a base class: `class MissingFile extends Gap.Base("MissingFile") { ... }`?
3. Should users use raw `Data.TaggedClass` directly with just a type constraint?

This affects type inference for resolution handlers - how does the resolution know what data fields are available on the gap it receives?

## Context

FR1.2.2.2 mentions explicit identity fields - suggesting gaps need to declare which fields contribute to identity. This implies some form of Gap definition API beyond raw `Data.TaggedClass`.

The resolution signature in FR1.3.1.1 is `(gap: Gap) => Effect<void, ResolutionError, R>` - but this generic `Gap` loses type information. Should it be `(gap: MissingFile) => Effect<...>` with type inference?

This directly impacts:
- How users define gap types (NFR2.3.4 TypeScript types)
- How resolutions are type-safe
- Whether gaps can declare identity fields (FR1.2.2.2)
