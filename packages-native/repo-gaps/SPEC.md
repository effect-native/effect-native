# Intent

Provide an automated system that maintains living documentation tracking:
1. **GAPS.md** — discrepancies between a package's SPEC.md and its implementation
2. **SPEC.QA.md** — open questions, ambiguities, and concerns about the spec itself

The system hooks into git's post-commit workflow, runs AI analysis in the background, and auto-commits updates without blocking the developer or creating infinite loops.

# Glossary

- **Package**: A directory containing a SPEC.md file that describes intended behavior
- **Gap**: A discrepancy between what the spec requires and what the implementation does
- **QA Item**: An open question, ambiguity, or concern about the specification itself
- **.ok directory**: A hidden directory within a package that contains hooks and logs
- **Auto-commit**: A commit generated by the system, marked with `[auto]` prefix

# Scope

## In-scope

- Post-commit hook that triggers gap analysis
- Walking up directory hierarchy to find `.ok/` directories for changed files
- AI-powered comparison of SPEC.md to implementation files
- AI-powered review of SPEC.md for ambiguities and open questions
- Tracking resolved vs open items across runs (preserving IDs)
- Auto-committing generated documentation
- Infinite loop prevention via commit message markers
- Concurrent execution of multiple package hooks
- Background execution (non-blocking)
- Logging to `.ok/commits.log`

## Explicit non-goals

- Pre-commit hooks or other git hook types
- Blocking the commit process
- Manual/interactive analysis modes
- Integration with CI/CD systems (beyond git hooks)
- Support for non-git version control
- Analysis of packages without SPEC.md

# End-State Behaviors

## B1 — Hook discovery via directory walking

When a commit includes changes to files within a package, the system finds all `.ok/` directories by walking up the directory tree from each changed file.

Observable:
- Changing `packages/foo/src/bar.ts` triggers hooks in `packages/foo/.ok/` if it exists
- Changing `packages/foo/sub/deep/file.ts` triggers hooks in all of: `packages/foo/sub/deep/.ok/`, `packages/foo/sub/.ok/`, `packages/foo/.ok/` (whichever exist)

## B2 — Concurrent background execution

Multiple `.ok/hooks/post-commit` scripts run concurrently in the background, not blocking the git commit from completing.

Observable:
- `git commit` returns immediately
- Hook output appears in `.ok/commits.log` asynchronously
- Multiple packages' hooks run in parallel

## B3 — Gap analysis updates GAPS.md

After each relevant commit, the AI compares SPEC.md to implementation files and updates GAPS.md with:
- Newly identified gaps
- Previously identified gaps (preserving IDs)
- Resolved gaps (moved to "Resolved" section with explanation)
- Severity and priority rankings

Observable:
- GAPS.md reflects current state of spec vs implementation
- GAP IDs persist across runs (GAP-1 remains GAP-1)
- Fixed issues move to "Resolved" section, not deleted

## B4 — Spec QA updates SPEC.QA.md

After each relevant commit, the AI reviews SPEC.md and updates SPEC.QA.md with:
- Ambiguities in the specification
- Underspecified behaviors
- Edge cases not addressed
- Suggested clarifications
- Resolved questions (moved to "Resolved" section)

Observable:
- SPEC.QA.md reflects current open questions about the spec
- QA IDs persist across runs
- Answered questions move to "Resolved" section

## B5 — Auto-commit with loop prevention

After updating GAPS.md and/or SPEC.QA.md, the system auto-commits these changes. Auto-commits are marked with `[auto]` prefix and skipped by subsequent hook runs.

Observable:
- Git log shows `[auto] Update GAPS.md and SPEC.QA.md` commits
- These commits do not trigger additional analysis
- No infinite loops occur

## B6 — Logging to commits.log

Each hook run logs its activity to `.ok/commits.log` with timestamp, commit SHA, and output.

Observable:
- `.ok/commits.log` contains timestamped entries
- Each entry shows what was analyzed and updated
- Errors are captured in the log

## B7 — Graceful handling of missing files

The system handles missing files gracefully:
- No SPEC.md → skip analysis
- No existing GAPS.md → create fresh analysis
- No existing SPEC.QA.md → create fresh analysis
- No implementation files → analyze spec only

Observable:
- New packages get initial GAPS.md and SPEC.QA.md on first commit
- System doesn't crash on missing optional files

# Layout

## Package structure
```
package/
├── SPEC.md              # Behavioral specification (human-authored)
├── GAPS.md              # Gap analysis (AI-generated)
├── SPEC.QA.md           # Spec QA (AI-generated)
├── src/                 # Implementation files
└── .ok/
    ├── hooks/
    │   └── post-commit  # Executable hook script
    └── commits.log      # Hook execution log
```

## Main hook location
```
repo/
├── bin/
│   └── post-commit      # Main entrypoint (installed to .git/hooks/)
└── .git/
    └── hooks/
        └── post-commit  # Symlink to bin/post-commit
```

# Interfaces

## Main post-commit hook (bin/post-commit)

Bash script that:
1. Gets list of changed files from `git diff-tree`
2. Walks up directory tree from each file looking for `.ok/` directories
3. Deduplicates found directories
4. Runs each `.ok/hooks/post-commit` in background
5. Logs output to `.ok/commits.log`

## Package hook (.ok/hooks/post-commit)

Bun/TypeScript script that:
1. Checks if last commit was auto-generated (skip if so)
2. Reads SPEC.md, implementation files, existing GAPS.md, existing SPEC.QA.md
3. Calls AI (OpenRouter) to analyze gaps and spec QA in parallel
4. Writes updated GAPS.md and SPEC.QA.md
5. Auto-commits changes with `[auto]` prefix

### Environment requirements
- `OPENROUTER_API_KEY` — API key for OpenRouter
- `bun` — Bun runtime available

### Exit codes
- 0: Success (or skipped auto-commit)
- Non-zero: Error (logged to commits.log)

# AI Prompt Contracts

## Gap analysis prompt

The AI receives:
- Full SPEC.md content
- Implementation file contents (src/*.ts)
- Existing GAPS.md (if any)

The AI produces:
- Markdown document with header "# GAPS.md — Gaps Between Spec and Implementation"
- "Resolved" section for previously-open items now fixed
- Sections for different gap categories
- Summary table with ID, Description, Severity, Status
- Preserved GAP IDs from previous analysis

## Spec QA prompt

The AI receives:
- Full SPEC.md content
- Existing SPEC.QA.md (if any)
- Implementation files (for context)

The AI produces:
- Markdown document with header "# SPEC.QA.md — Open Questions and Concerns"
- "Resolved" section for previously-open questions now answered
- Sections for Ambiguities, Underspecified Behaviors, Edge Cases
- Preserved QA IDs from previous analysis

# Performance Invariants

- Hook execution does not block git commit completion
- AI calls run in parallel (gaps + QA simultaneously)
- Multiple package hooks run concurrently
- Total analysis time bounded by AI response time (typically <60s)

# Security Considerations

- API keys read from environment, not hardcoded
- No sensitive data logged to commits.log
- Auto-commits only touch GAPS.md and SPEC.QA.md
- Hook scripts are local, not fetched remotely

# Compatibility

- Requires: git, bun, bash
- Works with: pnpm workspaces, git submodules
- AI provider: OpenRouter (configurable model)

# Open Design Freedoms

- Which AI model to use (currently claude-sonnet-4)
- Exact markdown format of generated files
- Which implementation files to include in analysis
- Whether to analyze test files
- Log rotation/cleanup strategy
